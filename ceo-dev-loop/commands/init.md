사용자가 `/ceo-dev-loop:init` 명령을 실행했습니다. 아래 작업을 순서대로 수행하세요.

## 입력
- 프로젝트 목표: $ARGUMENTS (비어있으면 사용자에게 먼저 물어볼 것)

## 할 일

### 0. 프로젝트 타입 감지

다음 중 **하나라도** 있으면 **기존 프로젝트(brownfield)** 로 간주, 전부 없으면 **greenfield** 로 간주:

- 패키지 매니페스트: `package.json`, `pyproject.toml`, `requirements.txt`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `Gemfile`, `composer.json`, `mix.exs`
- 코드 디렉토리: `src/`, `lib/`, `app/`, `cmd/`, `pkg/`
- 의미 있는 `README.md` (10줄 이상)

감지 결과를 사용자에게 **한 줄로 보고**: `감지됨: brownfield (React + TS)` 또는 `감지됨: greenfield (빈 디렉토리)`.

### 1. 기존 파일 확인
- 루트 `CLAUDE.md`가 있으면 → 내용 읽고, CEO-Dev 루프 섹션을 **추가**만 할 것 (기존 내용 유지)
- `docs/GOAL.md`, `docs/STATUS.md`가 있으면 → 덮어쓸지 사용자에게 확인

### 2. 프로젝트 스캔 (**brownfield 일 때만**)

다음을 순서대로 읽고 요약을 만든다. **토큰 낭비 방지** — 파일 하나당 최대 300줄, 스캔은 1분 내 완료 목표.

1. **README.md** — 프로젝트 설명, 빌드/실행/테스트 명령 추출
2. **패키지 매니페스트** — 언어·프레임워크·주요 의존성·스크립트
3. **디렉토리 구조** — `Glob` 으로 루트 1단계 + 주요 코드 디렉토리(`src/`, `lib/`, `app/` 등) 2단계 파악
4. **핵심 설정 파일** (있는 것만): `tsconfig.json`, `.eslintrc*`, `vitest.config.*`, `jest.config.*`, `Dockerfile`, `docker-compose.yml`, `.github/workflows/*`
5. **기존 `docs/`** — 문서 목록만 파악 (내용 전체 읽기 금지)

요약 항목 (GOAL.md 의 `## 현재 프로젝트 맥락` 섹션에 채워 넣을 내용):
- **기술 스택** — 언어 + 프레임워크 + 주요 라이브러리 (5개 이내)
- **주요 모듈** — 경로 + 한 줄 역할 (최대 8개)
- **빌드/개발/테스트 명령** — 매니페스트 스크립트 또는 README 에서 추출
- **기존 인터페이스/API** — 공개 엔드포인트, export 된 주요 타입/함수 (있으면)
- **CI/배포 힌트** — 워크플로우 파일에서 추출 (있으면)

불확실한 정보는 적지 말고 "확인 필요"로 표시.

### 3. `docs/` 폴더 생성 및 파일 작성

#### `docs/GOAL.md` — **brownfield 버전**

```markdown
# 프로젝트 목표

## 최종 목표
$ARGUMENTS

## 현재 프로젝트 맥락
### 기술 스택
(스캔 결과)

### 주요 모듈
- `경로/` — 역할
- ...

### 빌드/개발/테스트
- 개발: (명령)
- 빌드: (명령)
- 테스트: (명령)

### 기존 인터페이스/API (보존 대상)
(있으면 나열, 없으면 "특별히 없음")

### CI/배포
(있으면 한 줄, 없으면 생략)

## 필수 요구사항
(목표를 구체적 작업으로 분해한 1차 초안 — Claude 가 제시, 사용자가 보강/승인)

- [ ] ...
- [ ] ...

> **phase/milestone 으로 나뉜 프로젝트라면** 위 단순 리스트 대신 `### Phase 1 — 이름` / `### Phase 2 — 이름` 헤더로 묶고 **모든 phase 의 요구사항을 빠짐없이 체크박스로 미리 작성**한다. 일부만 적으면 CEO 가 그 부분 완료 시 `[DONE]` 으로 오판해 자율 루프가 끊긴다.

## 제약 조건
- 기술 스택: (스캔에서 탐지한 스택 유지)
- 기존 공개 API/인터페이스 보존
- 사용 금지: (사용자 추가)

## 완료 기준 (DoD)
- [ ] 빌드 성공
- [ ] 모든 필수 요구사항 구현
- [ ] 기존 테스트 유지 통과
- [ ] 기본 동작 검증
```

### 4. `CLAUDE.md`에 루프 규칙 추가 (없으면 새로 만들고, 있으면 아래 섹션만 append)

````markdown

## CEO-Dev 자동 루프 규칙 (ceo-dev-loop v0.3.3)

당신은 이 프로젝트의 메인 Dev 에이전트입니다. 아래 흐름을 엄격히 따르세요.

### 작업 흐름
1. `docs/GOAL.md`에서 목표와 "현재 프로젝트 맥락" 확인
2. 코드 작성 / 파일 생성
3. `docs/STATUS.md` 갱신 — 진척, 최근 작업, **활성 컨텍스트**, 턴 카운터 +1
4. `@ceo` 호출 → 검토 및 다음 지시 수신
5. CEO 응답 처리:
   - `[DONE]` → 종료 절차
   - `[COMPACT]` 또는 `[CLEAR]` 신호 있음 → 컨텍스트 리프레시 절차 (아래)
   - 그 외 → CEO 지시에 따라 2번으로 반복

### 첫 턴 규칙 (brownfield)
`docs/GOAL.md` 에 "현재 프로젝트 맥락" 섹션이 있으면 (= brownfield) 첫 작업 시작 전 반드시:
1. "현재 프로젝트 맥락" 숙지
2. 첫 작업과 **직접 관련된 모듈 파일**을 `Read` 로 선행 확인
3. 그 위에서 구체적 작업 계획을 세워 `@ceo` 에게 첫 호출

### @ceo 호출이 **반드시** 필요한 시점
- 기능 하나/파일 하나/리팩터 하나 단위 완료 시
- 라이브러리 선택, 아키텍처 분기, 트레이드오프 발생 시
- 요구사항이 모호하거나 방향이 불확실할 때
- 빌드 성공 / 테스트 통과 후 다음 단계 확인 시
- **5턴 누적 시** (자체 판단으로도 호출해 컨텍스트 점검 받기)
- **brownfield**: 기존 공개 API/인터페이스를 변경해야 할 때

### 호출 형식
```
@ceo 검토 요청

[완료한 작업]
- 생성/수정한 파일
- 핵심 변경 내용

[현재 상태]
- 빌드/실행 상태
- 알려진 이슈
- 마지막 컴팩트 이후 N 턴 (STATUS.md 턴 카운터)

[결정 필요 사항] (있을 때만)
- 옵션 A vs B
```

### 컨텍스트 리프레시 절차 (v0.3.0 신규)

CEO 응답에 `[COMPACT]` 또는 `[CLEAR]` 가 있으면 **CEO 지시 작업을 수행하기 전에** 다음을 먼저:

1. `docs/STATUS.md` `## 활성 컨텍스트` 섹션을 재시작 가능 수준으로 **풍부하게** 보강
   - 현재 만지는 파일 경로 + 어디까지 했는지
   - 미해결 결정/가설
   - 다음 액션의 첫 단계
2. 새로 내려진 결정은 `docs/DECISIONS.md` 에 누적
3. milestone 완료 직후라면 `docs/checkpoints/NN-{slug}.md` 스냅샷 작성 (선택)
4. `/compact` 또는 `/clear` 실행
5. 재진입 후 첫 액션:
   - `docs/GOAL.md` → `docs/STATUS.md` → `docs/DECISIONS.md` 순으로 다시 읽기
   - STATUS.md 의 턴 카운터 0 으로 리셋
   - CEO 가 마지막에 지시한 작업 수행

### 종료 조건
CEO 응답이 `[DONE]`으로 시작하면:
1. `docs/STATUS.md` 최종 상태 기록 (진척 N/N, 활성 컨텍스트는 "완료")
2. 작업 중단
3. 사용자에게 최종 요약 보고

### 자율 루프 유지 (v0.3.1 — 회귀 방지)
사용자가 `/ceo-dev-loop:start` 또는 "알아서 진행" 류 지시를 한 시점부터 **GOAL.md 끝까지** 자율 권한이 유효합니다. milestone/phase 경계에서 사용자 입력 대기 모드로 빠지지 마세요.

CEO 응답이 `[DONE]` 이 아니면 (= 일반 지시 또는 `[COMPACT]/[CLEAR]` 신호) **반드시** 다음 작업을 즉시 시작합니다. 다음 표현으로 응답을 끝내지 말 것:
- "다음 작업 지시를 기다립니다"
- "Phase N 진행하시려면 알려주세요"
- "사용자 승인 후 진행"

CEO 가 모호하게 끝내거나 `[결정 / 다음 작업]` 이 비어 있으면 **즉시 `@ceo` 재호출** 해서 구체적 지시를 받아내야 합니다 — 사용자에게 묻지 말 것.

### `[DONE]` 수신 시 Dev 의 1차 필터 (v0.3.2 — 안전벨트)
CEO 가 `[DONE]` 을 반환해도 Dev 는 **종료하기 전에** 아래를 직접 확인합니다:

1. `docs/GOAL.md` 의 `## 최종 목표` 문장을 한 번 더 읽기
2. 목표에 "phase" / "milestone" / "단계" / "N.M" / 참조 ADR 언급이 있으면, `## 필수 요구사항` 체크박스가 **최종 phase 까지 모두 포함**되는지 확인
3. 누락된 phase 가 감지되면 종료하지 않고 **즉시 `@ceo` 에게 `[GOAL 완전성] 의심` 재호출**:
   ```
   @ceo [DONE] 수신했으나 GOAL.md 재검토 결과 phase 1.X 이후 요구사항 누락 의심.
   최종 목표: "..."
   현재 체크박스: phase 1~K 만 커버
   GOAL.md 보강 또는 진짜 DONE 재판정 부탁합니다.
   ```
4. CEO 가 보강 지시를 주면 Dev 가 GOAL.md 수정 → 다시 `@ceo` 호출 → 새 `[DONE]` 받을 때만 실제 종료

이 1차 필터는 CEO 의 `[DONE]` 오발을 한 번 더 막는 안전벨트. 사용자에게 묻지 말 것 — Dev 스스로 검증.

### 예외 (사용자 확인 필수)
- `git push`, 배포
- 외부 API 키/시크릿 필요 작업
- 파괴적 작업 (파일 삭제 대량, DB 마이그레이션)
- **brownfield**: 공개 API/인터페이스 breaking change
- 같은 작업이 2회 연속 실패하여 CEO 가 접근 변경 또는 사용자 확인을 지시한 경우

그 외에는 사용자 확인 없이 CEO 승인으로 진행.
````
