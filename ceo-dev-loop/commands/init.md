사용자가 `/ceo-dev-loop:init` 명령을 실행했습니다 (ceo-dev-loop v0.4.1). 아래 작업을 순서대로 수행하세요.

> **v0.4.0 핵심 원칙**
> - 사용자 승인은 **2가지 안전장치 외엔 묻지 않는다**: ① 기존 `docs/GOAL.md` 덮어쓰기 ② CEO 가 "정보 부족 — 사용자 답변 필요" 신호를 줬을 때
> - GOAL.md 의 분해·DoD·진행 매핑·phase 판단은 **모두 CEO 가 결정**. Dev 는 컨텍스트 수집 + 파일 IO + CEO 호출만
> - init 끝나면 즉시 `/start` 흐름으로 자연 전환 (별도 명령 불필요)

## 입력
- 프로젝트 목표: $ARGUMENTS (비어있으면 사용자에게 한 번만 물어볼 것)

## 1. 프로젝트 성숙도 4단계 감지

다음 두 축으로 분류:

**축 1 — 코드 존재 여부:**
- 패키지 매니페스트: `package.json`, `pyproject.toml`, `requirements.txt`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `Gemfile`, `composer.json`, `mix.exs`
- 코드 디렉토리: `src/`, `lib/`, `app/`, `cmd/`, `pkg/`
- 의미 있는 `README.md` (10줄 이상)

**축 2 — 진행 문서 존재 여부 (Glob 으로 빠르게 검색):**
- `docs/GOAL.md`, `docs/STATUS.md` (= 본 플러그인의 완전 문서)
- `docs/PROGRESS*.md`, `docs/ROADMAP*.md`, `docs/PHASE_*.md`, `docs/PHASES.md`
- `docs/ADR*/**/*.md`, `docs/DECISIONS*/**/*.md`
- `TASKS.md`, `BACKLOG.md`, `TODO.md`, `RFC*.md`
- `README.md` 안의 "Roadmap" / "Status" / "TODO" 섹션 (있으면)

**분류 결과:**

| 단계 | 코드 | 진행 문서 |
|------|------|-----------|
| **A. Greenfield** | 없음 | 없음 |
| **B. Brownfield 무문서** | 있음 | 없음 |
| **C. Brownfield 부분문서** | 있음 | 산재 (GOAL.md 는 없음) |
| **D. Brownfield 완전문서** | 있음 | `docs/GOAL.md` 존재 |

감지 결과를 한 줄로 보고: `감지됨: 단계 C (React + TS, ROADMAP.md + ADR 5개)`.

## 2. 안전장치 — D 단계 덮어쓰기 확인 (사용자 입력 1회 — 유일한 필수 확인)

**D 단계만** 사용자에게 한 번 묻기:
```
docs/GOAL.md 가 이미 존재합니다. 어떻게 할까요?
1) 합류 — 기존 GOAL 그대로 두고 STATUS/DECISIONS 만 보강 (default)
2) 덮어쓰기 — 새 GOAL.md 를 CEO 가 다시 분해해서 작성 (기존은 docs/GOAL.md.bak 으로 백업)
```

답변 없으면 default(합류). 합류 모드면 이후 단계 3은 건너뛰고 4부터.

A/B/C 단계는 묻지 않는다.

## 3. 컨텍스트 수집 (단계별)

### B/C/D — 코드 스캔 (토큰 절약: 파일당 최대 300줄, 1분 내 완료)
1. README.md — 프로젝트 설명, 빌드/실행/테스트 명령
2. 패키지 매니페스트 — 언어·프레임워크·주요 의존성·스크립트
3. 디렉토리 구조 — Glob 으로 루트 1단계 + 주요 코드 디렉토리(`src/`, `lib/`, `app/`) 2단계
4. 핵심 설정 (있는 것만): `tsconfig.json`, `.eslintrc*`, `vitest.config.*`, `jest.config.*`, `Dockerfile`, `docker-compose.yml`, `.github/workflows/*`

### C/D — 진행 문서 흡수 (파일당 최대 100줄)
단계 1 에서 발견한 진행 문서를 **모두** 빠르게 읽어 다음 추출:
- 현재 phase / milestone (있으면)
- 완료된 작업 목록
- 진행 중 / 미완 작업 목록
- 주요 결정 이력 (ADR / DECISIONS)

불확실한 정보는 적지 말고 "확인 필요"로 표시.

## 4. CEO 에게 GOAL 분해 위임 (v0.4.0 핵심)

위에서 모은 컨텍스트를 패키지로 만들어 `@ceo init:` 호출:

```
@ceo init: GOAL.md 1차 초안 분해 요청

[사용자 목표]
{$ARGUMENTS}

[프로젝트 단계]
{A | B | C | D — 한 줄 근거}

[코드 스캔 결과]  (B/C/D 만)
- 스택: ...
- 주요 모듈: ...
- 빌드/테스트 명령: ...
- CI/배포: ... (있으면)

[진행 문서 흡수]  (C/D 만)
- {경로}: {추출 핵심 N줄}
- ...

[요청]
agents/ceo.md 의 "init 단계 GOAL 분해 요청 처리" 응답 형식대로 회신.
정보 부족은 진짜 분해 불가능한 핵심에만 한정 — 합리적 추정으로 채울 수 있는 건 채워서 자율 진행 권한 부여까지 한 번에.
```

CEO 응답을 받아 다음 처리:

### 4a. CEO 응답에 `[정보 부족] 없음` 이고 `[자율 진행 권한 부여] 예` 면 → 단계 5로 직진 (사용자 입력 0)

### 4b. `[정보 부족]` 에 항목이 있으면 → 사용자에게 그 질문들만 한 번에 묶어 묻기 → 답변 받고 같은 패키지에 추가해 `@ceo init:` 재호출 (최대 1회 추가)

### 4c. 재호출 후에도 `[정보 부족]` 이 남으면 → 그 부분만 "TBD — 사용자 추후 확정" 으로 GOAL.md 에 표시하고 진행 (멈추지 않는다)

### `CLAUDE.md` (없으면 새로 만들고, 있으면 아래 섹션만 append)

````markdown

## CEO-Dev 자동 루프 규칙 (ceo-dev-loop v0.4.1)

당신은 이 프로젝트의 메인 Dev 에이전트입니다. 아래 흐름을 엄격히 따르세요.

### 작업 흐름
1. `docs/GOAL.md`에서 목표와 "현재 프로젝트 맥락" 확인
2. 코드 작성 / 파일 생성
3. `docs/STATUS.md` 갱신 — 진척, 최근 작업, **활성 컨텍스트**, 턴 카운터 +1
4. `@ceo` 호출 → 검토 및 다음 지시 수신
5. CEO 응답 처리:
   - `[DONE]` → 종료 절차 (단, 아래 "Dev 1차 필터" 먼저)
   - `[COMPACT]` 또는 `[CLEAR]` 신호 있음 → 컨텍스트 리프레시 절차
   - 그 외 → CEO 지시에 따라 2번으로 반복

### 첫 턴 규칙 (brownfield)
`docs/GOAL.md` 에 "현재 프로젝트 맥락" 섹션이 있으면:
1. "현재 프로젝트 맥락" 숙지
2. 첫 작업과 직접 관련된 모듈 파일을 `Read` 로 선행 확인
3. 그 위에서 구체적 작업 계획을 세워 `@ceo` 에게 첫 호출

### @ceo 호출이 반드시 필요한 시점
- 기능 하나/파일 하나/리팩터 하나 단위 완료 시
- 라이브러리 선택, 아키텍처 분기, 트레이드오프 발생 시
- 요구사항이 모호하거나 방향이 불확실할 때
- 빌드 성공 / 테스트 통과 후 다음 단계 확인 시
- 5턴 누적 시 (자체 판단으로도 호출해 컨텍스트 점검 받기)
- brownfield: 기존 공개 API/인터페이스를 변경해야 할 때

### 호출 형식
```
@ceo 검토 요청

[완료한 작업]
- 생성/수정한 파일
- 핵심 변경 내용

[현재 상태]
- 빌드/실행 상태
- 알려진 이슈
- 마지막 컴팩트 이후 N 턴 (STATUS.md 턴 카운터)

[결정 필요 사항]  (있을 때만)
- 옵션 A vs B
```

### 컨텍스트 리프레시 절차
CEO 응답에 `[COMPACT]` 또는 `[CLEAR]` 가 있으면 CEO 지시 작업 수행 전에 먼저:
1. `docs/STATUS.md` `## 활성 컨텍스트` 섹션을 재시작 가능 수준으로 보강
2. 새 결정은 `docs/DECISIONS.md` 에 누적
3. milestone 완료 직후라면 `docs/checkpoints/NN-{slug}.md` 스냅샷 (선택)
4. `/compact` 또는 `/clear` 실행
5. 재진입 후: GOAL → STATUS → DECISIONS 다시 읽기 + 턴 카운터 0 리셋 + CEO 마지막 지시 수행

### 종료 조건
CEO 응답이 `[DONE]` 으로 시작하면:
1. `docs/STATUS.md` 최종 상태 기록 (진척 N/N, 활성 컨텍스트는 "완료")
2. 작업 중단
3. 사용자에게 최종 요약 보고

### 자율 루프 유지 (핵심)
사용자가 init 또는 `/ceo-dev-loop:start` 또는 "알아서 진행" 류 지시를 한 시점부터 **GOAL.md 끝까지** 자율 권한이 유효합니다. milestone/phase/세션/컴팩트 경계에서 사용자 입력 대기 모드로 빠지지 마세요.

CEO 응답이 `[DONE]` 이 아니면 **반드시** 다음 작업을 즉시 시작합니다. 다음 표현으로 응답을 끝내지 말 것 (v0.4.1 어휘 확장):

**phase 경계 우회:**
- "다음 작업 지시를 기다립니다"
- "Phase N 진행하시려면 알려주세요"
- "사용자 승인 후 진행"

**세션/컨텍스트 우회 (v0.4.1 신규):**
- "세션 종료 보고" / "본 세션 성과"
- "다음 세션 재진입 시" / "다음 세션에서"
- "/ceo-dev-loop:start ... 입력 시 즉시 ... 착수"
- "단일 세션 내 ~ 물리적 제약"

CEO 가 모호하게 끝내거나 `[결정 / 다음 작업]` 이 비어 있으면 **즉시 `@ceo` 재호출** 해서 구체적 지시를 받아내야 합니다 — 사용자에게 묻지 말 것.

### Dev 측 컨텍스트 한계 단계적 처리 (v0.4.1 — 인프라 영역)

CEO 는 인프라 판단 권한이 없습니다 (ceo.md "CEO 권한 범위" 참조). 컨텍스트 한계 처리는 **Dev 가 단계적으로** 수행합니다:

1. **1차** — CEO 가 `[COMPACT]` 신호를 주면 위 "컨텍스트 리프레시 절차" 따름. 같은 세션에서 이어감
2. **2차** — `[COMPACT]` 후에도 컨텍스트 압박이 계속되면 다음 호출에서 `[CLEAR]` 신호를 받거나 Dev 가 자체 판단으로 `/clear` 실행 (전제: STATUS/DECISIONS 매우 상세히 보강 후). 같은 세션에서 GOAL/STATUS/DECISIONS 재독으로 이어감
3. **3차** — 1, 2차 모두 한계면 그제서야 사용자 에스컬레이션. STATUS.md 에 `## 인프라 한계 — 새 세션 재진입 가이드` 섹션 작성 후 사용자에게 한 번 보고

**1차/2차 단계에서 "세션 종료/다음 세션/재진입" 같은 표현으로 보고 종료 절대 금지.** 컨텍스트 한계는 같은 세션 내 컴팩트/클리어로 처리되는 것이 default.

### `[DONE]` 수신 시 Dev 1차 필터 (안전벨트)
CEO 가 `[DONE]` 을 반환해도 종료 전에 직접 확인:
1. `docs/GOAL.md` 의 `## 최종 목표` 다시 읽기
2. 목표에 phase/milestone/단계/N.M/ADR 언급이 있으면, `## 필수 요구사항` 체크박스가 최종 phase 까지 모두 포함되는지 확인
3. 누락 의심 시 즉시 `@ceo` 에게 `[GOAL 완전성] 의심` 재호출
4. CEO 가 진짜 DONE 재판정 또는 보강 지시 → 보강 후 다시 `@ceo` → 새 `[DONE]` 받을 때만 실제 종료

### CEO 응답 우회 어휘 감지 시 즉시 재호출 (v0.4.1 — 안전벨트 확장)

CEO 응답이 `[DONE]` 도 `[COMPACT]/[CLEAR]` 도 아닌데, 본문에 다음 어휘 중 하나라도 있으면 **사용자에게 출력하지 말고 즉시 `@ceo` 재호출**:

- "세션 종료 보고" / "본 세션 성과"
- "다음 세션 재진입 시" / "다음 세션에서"
- "/ceo-dev-loop:start ... 입력 시" / "{문구} 입력 시 즉시 ... 착수"
- "물리적 제약" / "단일 세션 내" / "재진입 가능"
- "다음 작업 지시 바랍니다" / "Phase N 진행하시려면" / "사용자 승인 후 진행"

재호출 메시지:
```
@ceo 직전 응답에 우회 어휘 감지 (예: "세션 종료/재진입 시").
v0.4.1 ceo.md "자율 루프 권한" + "CEO 권한 범위" 규약 위반.
컨텍스트 한계는 [COMPACT]/[CLEAR] 신호로만 표현 가능. 인프라 판단 금지.
[목표 진척] 박고 다음 작업 즉시 지시 또는 명확한 [COMPACT] 신호로 재응답 부탁합니다.
```

이 안전벨트는 v0.4.1 이전 버전 CEO 가 구버전 학습된 패턴으로 우회 시도를 해도 한 번 더 차단합니다.

### 예외 (사용자 확인 필수 — 이것만)
- `git push`, 배포
- 외부 API 키/시크릿 필요 작업
- 파괴적 작업 (파일 삭제 대량, DB 마이그레이션)
- brownfield: 공개 API/인터페이스 breaking change
- 같은 작업이 2회 연속 실패하여 CEO 가 접근 변경 또는 사용자 확인을 지시한 경우

그 외에는 사용자 확인 없이 CEO 승인으로 진행.
````

## 6. 완료 보고 + 자동 자율 루프 시작 (사용자 승인 0회)

```
✅ CEO-Dev 루프 v0.4.1 설치 완료 (단계 {A | B | C | D})

GOAL.md 요약 (CEO 1차 초안):
- 최종 목표: ...
- 필수 요구사항: N개 ({phase 구조 또는 단순 리스트})
- DoD: M개
- 기존 진행 매핑: K/N 완료  (C/D 만)

자율 루프 시작합니다. 멈추려면:
- 즉시 다른 메시지 입력
- docs/GOAL.md 직접 편집 후 /ceo-dev-loop:start 재호출

지금부터 CEO 첫 지시에 따라 작업 시작.
```

보고 직후 **사용자 입력 없이 즉시** `/ceo-dev-loop:start` 의 본문 절차를 그대로 수행:

1. `docs/GOAL.md`, `docs/STATUS.md`, `docs/DECISIONS.md` 다시 읽기 (방금 작성했으므로 캐시 그대로 OK)
2. `@ceo` 호출 — 시작 메시지:
   ```
   @ceo 프로젝트 시작 (init 직후 자동 진입)

   [목표]
   (GOAL.md 핵심 요약 + 진척 K/N)

   [현재 상태]
   - {C/D 면 진행 매핑 결과 / A/B 면 "초기 상태"}
   - 활성 컨텍스트: (init 직후, 비어있음)
   - 마지막 컴팩트 이후 0 턴

   [요청]
   첫 작업 지시 바랍니다 (파일/함수 + 검증 명령).
   ```
3. CEO 응답 받아 CLAUDE.md 규칙대로 루프 진행
4. `[DONE]` 까지 자동 반복 (Dev 1차 필터 적용)
